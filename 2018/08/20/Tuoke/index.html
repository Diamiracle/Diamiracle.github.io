1.简介

以前都是看一些脱壳的文章，这次决定动手试试脱一下，下面总结一下整个脱壳过程。

脱壳样本为AliCTF2014evilAPK400，之前早有大神发过脱壳方法，所以不懂的地方就去看看大神的文章，这样的学习起来就不会觉得无助了。

2.步骤

1）查壳

AndroidKiller载入APK，提示该APP加壳了，xml中application类里有

android:name="com.ali.mobisecenhance.StubApplication"

一般加壳程序都会在application类里声明“android:name”，然后加载还原Dex的so库，此题也一样，类里加载了libmobisec.so，并且声明了三个native函数“b”，“attachBaseContext”和“onCreate”，其他的就没什么信息了。

2）动态调试

于是动态调试，由于没什么信息，就直接在调试的时候在libmobisec.so里搜索dex，发现几个很可疑的函数“parsedex”、“dexjuicerpatch”等，其实这个壳只是最初的雏形，所以没加什么混淆，这对于我们初学者来说是很好的学习样本。接着在这几个可疑的函数都下断，F9运行，断在了parsedex函数里，然后就是一步一步的跟代码了。调试代码的时候可以把汇编代码转成伪代码来调试，这样可以加快分析的进度。

程序首先打开assets目录下的cls.jar文件进行脱壳，函数为parse_dex里的openWithHeader()函数：

v32 = ((int (*)(ali::EncFile *, unsigned __int8 **, unsigned int *, unsigned int))ali::EncFile::openWithHeader)(
          v31,
          (unsigned __int8 **)&v58,
          (unsigned int *)&v57,
          0x10u);

进去后里面有很多log，这也对分析提供了很多额外信息。整个脱壳过程为：

cls.jar->decryptRc4()解密->Lzma解压
Rc4的密钥为0x18字节：

RC4_key:88 54 EB E8 A3 7E F2 F8  9F 75 F8 A6 8F 51 AB A0 98 33 C8 A8 8B 7C D0 94  65 61 64 00 89 3D 01 00

Lzma解压后就可以dump出dex了，但是这里dump出的dex是不正确的，因为后面还有用同样的方法将assets目录下的juice.data解密出来对它进行patch的。原因在大神的文章里说了，此时dump出来的dex文件的method是被统一改成了同一个函数体：

new-instance v0, Ljava/lang/RuntimeException;

invoke-direct {v0}, Ljava/lang/RuntimeException;-><init>()V

这里注意dump时的文件大小，解压的时候文件是增大了的。然后找到juice.data解密的地方：

((_DWORD (*)(ali *, unsigned __int8 *, unsigned int, const char *))ali::dex_juicer_patch)(v33, v57, v101, v52)

里面的open()函数：

v6 = ((int (*)(ali::EncFile *, unsigned __int8 **, unsigned int *))ali::EncFile::open)(
     v5,
     (unsigned __int8 **)&dword_752550E4[69],
     (unsigned int *)&dword_752550E4[68]);

然后又进入到openWithHeader()函数进行解密。dump出解密的juice.data文件。

3）指针修复

最后就是修复了，这里可以不用像大神那样去还原dex，可以直接在程序执行到patch结束时dump出cls.dex+decrypt.juice.data的内存。中间会有很多0x00填充，只要计算出文件的大小dump就可以了。刚到这里的时候，为了找CodeOff也是花了很多时间，以前也没怎么详细的去跟过dex的二进制，这次全补回来了。哈哈哈！最后发现可以直接把cls.dex+decrypt.juice.data的内存dump出来，因为decrypt.juice.data就接在cls.dex后面，因为cls.dex只是CodeOff被改了，这里只是把CodeOff还原出来，其他的没变。

3.脱壳完成

当然大神还是大神，直接把还原时的算法给逆向出来了。最后就是要使dump出来的dex文件能被反编译为java了，但是直接用jeb和其他成型的工具都是该dex为无效的dex。这里也是坑了我很久，原来是使用工具的问题，反编译时用到的是baksmali和dex2jar，不要用其他的集成工具，所以分析时工具要用对啊，不然做好久都做不出来。

首先用baksmali.jar反编译recovery.dex（也就是patch之后dump出来的dex），可以看到method都已经还原了。

.method static synthetic a(Lcom/ali/encryption/MainActivity;)V
    .registers 2

    invoke-static {}, Landroid/support/v4/app/qn;->a()Z

    move-result v0

    invoke-static {v0}, Landroid/support/v4/app/qn;->b(I)V

    invoke-direct {p0}, Lcom/ali/encryption/MainActivity;->a()V

    return-void
.end method

然后用dex2jar反编译recovery.dex出错，删掉反编译出来的所有的testdex2jarcrash()函数，然后用smali.jar将smali文件重新打包成dex文件，再用dex2jar反编译就可以了。
